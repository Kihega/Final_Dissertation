  REVISION NOTE (v2.0)
  ─────────────────────────────────────────────────────────────────────────
  All citizen data, facility data, and civil registry records are stored
  and managed entirely within the NBS Central Database. There are NO
  external data-source dependencies for citizen lookups. National IDs are
  auto-generated by the system at birth registration. All personal data
  is entered at the appropriate time by the officer with the correct role.

  External services are limited to:
    • Email / SMS delivery (notifications, OTP)
    • Government PKI (certificate signing only)

  The RITA civil registry module is an INTERNAL service within the
  NBS system, not an external API endpoint.
  ─────────────────────────────────────────────────────────────────────────

  2.1  SUPER ADMIN ONBOARDING FLOW
────────────────────────────────────────────────────────────────────────────────

  [Existing Super Admin Dashboard]
          │
          ▼
  STEP 1: Enter New Admin's Details (Manual Entry)
          │  • Employee ID
          │  • Full name
          │  • Date of birth
          │  • Gender
          │  • National ID number
          │    → System verifies National ID exists in internal citizens table
          │    → Confirms age ≥ 18 and vital_status = ALIVE
          │  • Official email address
          ▼
  STEP 2: Internal Citizen Record Lookup
          │  • System queries citizens table by national_id
          │  • Auto-fills: full name, DOB, gender (from internal DB)
          │  • Displays citizen record for admin to confirm — no external API
          ▼
  STEP 3: Generate & Send Login Token
          │  • System creates signed one-time JWT (24h TTL)
          │  • SHA-256 hash stored in DB
          │  • Official email sent with:
          │    - Login token
          │    - Super Admin dashboard URL
          ▼
  STEP 4: New Admin Opens Dashboard Link
          │  • Lands on Welcome / Authorization page
          │  • Enters one-time token
          │  • System validates token hash + expiry
          ▼
  STEP 5: Profile Setup Page
          │  • New password (min 12 chars, complexity rules)
          │  • Confirm password
          │  • Upload or capture profile photo
          │  • Valid mobile number
          │  • Department within NBS
          ▼
  STEP 6: Account Activated
          │  • Token invalidated immediately after use
          │  • Account status set to ACTIVE
          ▼
  STEP 7: Login with Email + Password
          │  • Optional MFA (TOTP app or SMS OTP)
          │  • JWT issued on success
          ▼
  [Super Admin Dashboard Access]

  2.6  CITIZEN REGISTRATION FLOW  (by Village Officer — for citizens aged 18+)
────────────────────────────────────────────────────────────────────────────────

  CONTEXT: Every citizen already has a record in the citizens table from birth
  registration (Section 2.7). The village officer is completing that record with
  full personal, biometric, and socio-economic details at age 18.

  [Village Officer App — inside village geo-fence]
          │
          ▼
  STEP 1: Enter National ID Number
          │  • System queries internal citizens table by national_id
          │  • Validates citizen age ≥ 18 (computed from DOB in DB)
          │  • Validates vital_status = ALIVE
          │  • Rejects if under 18 or deceased
          ▼
  STEP 2: Internal Citizens Table Lookup
          │  • Auto-populate from citizens table:
          │    name, DOB, gender, father_citizen_id, mother_citizen_id
          │  • Check disability_status flag in internal record
          │    - If disability_status = true → ask officer:
          │      "Still disabled?" or "Cured?"
          │      (allow selecting cured disabilities for multi-disabled)
          ▼
  STEP 3: Biometric Capture
          │  • Take live photo (camera)
          │  • Capture fingerprints (hardware scanner)
          │  • Capture signature (touchscreen)
          │  • Stored encrypted in internal DB / object storage
          ▼
  STEP 4: Extended Personal Details
          │  • Disability type(s): Hearing / Sight / etc. (multi-select)
          │  • Disability cause: Road accident / Born with / etc.
          │  • Education level (select one):
          │    None / Primary / O-Level / A-Level / Certificate /
          │    Diploma / Bachelor's / Master's / PhD / Other (VETA, online)
          │  • Village street name
          │  • House registration number
          │  • Marital status (Yes/No)
          │    - If Yes → recommend marriage registration (optional, consent)
          │  • Children National IDs (add multiple, each validated in citizens table)
          │  • Occupation (primary + additional economic activities)
          ▼
  STEP 5: Optional / Supplementary Details
          │  • Land ownership in acres (optional)
          │  • Animal keeping category (multi-select):
          │    Livestock (cows/goats/pigs) / Birds (hens) / Other (dogs/donkeys)
          │  • Health insurance provider(s) (multi-select)
          │  • Communication devices (multi-select):
          │    Mobile (kitochi / smartphone), Computer, Radio, Television
          │  • Social Mutual Fund memberships (multi-select):
          │    NSSF / TASAF / etc.
          ▼
  STEP 6: Review & Confirm
          │  • Officer reviews all captured data
          │  • Citizen confirms (verbal or signature)
          ▼
  STEP 7: National ID Card Issued
          │  • Citizens table record UPDATED with full profile
          │  • national_id_issued = today's date
          │  • national_id_expires = today + 10 years
          │  • ID card print payload generated from internal record
          │  • After 10 years: citizen returns for photo, fingerprint,
          │    and signature renewal if vital_status = ALIVE
          ▼
  [Citizen Record Fully Populated in Central Database]

  2.7  BIRTH REGISTRATION FLOW  (by Hospital Officer)
────────────────────────────────────────────────────────────────────────────────

  THIS IS WHERE THE NATIONAL ID IS BORN — the first record for any citizen.

  [Hospital Officer App — inside facility geo-fence]
          │
          ▼
  STEP 1: Enter Birth Details
          │  • Child first name, middle name, surname
          │  • Gender (Male / Female)
          │  • Date of birth
          ▼
  STEP 2: Link Parents
          │  • Father National ID → system validates in internal citizens table
          │  • Mother National ID → system validates in internal citizens table
          │  • System fetches parent names and links child to lineage records
          ▼
  STEP 3: Link Facility & Officer to Child Record
          │  • Facility ID auto-attached from officer's profile (internal DB)
          │  • Officer (doctor/nurse) citizen_id linked from officer profile
          ▼
  STEP 4: Generate National ID for Newborn
          │  • System AUTO-GENERATES unique national_id for newborn
          │    (format: NBS-generated unique sequential code, e.g. TZ-YYYYMMDD-XXXXX)
          │  • NEW record inserted into internal citizens table
          │  • Child vital_status = ALIVE
          │  • Age tracking starts from DOB
          │    (auto-increments every year while vital_status = ALIVE)
          ▼
  STEP 5: Issue Birth Certificate (Internal)
          │  • Birth record inserted into births table in central DB
          │  • PDF certificate generated
          │  • QR code embedded (digitally signed via govt PKI)
          │  • Certificate stored in internal rita_certificates table
          │  • Individual receives printed/digital copy
          ▼
  NOTE: When child turns 18 → must visit Village Officer
        to complete full citizen profile (Section 2.6) and receive
        National ID card. The national_id already exists since birth.

  2.8  DEATH REGISTRATION FLOW
────────────────────────────────────────────────────────────────────────────────

  CASE A — Death OUTSIDE hospital (Village Officer registers):
          │
          ▼
  • Enter citizen National ID
  • System fetches citizen info from internal citizens table
  • Enter: date of death, cause / reason of death
  • Citizen vital_status updated to DECEASED in citizens table
  • Age counter frozen at last recorded age
  • Death certificate generated (PDF + QR, stored in internal DB)
  • Individual / family receives printed or digital copy


  CASE B — Death INSIDE hospital (Hospital Officer registers):
          │
          ▼
  • Select category: INFANT or ADULT
  │
  ├─► If INFANT:
  │     • Enter father National ID → validate in internal citizens table
  │     • Enter mother National ID → validate in internal citizens table
  │     • Date of death
  │     • Cause / reason (if applicable)
  │     • Infant death record created in deaths table
  │     • Certificate generated → stored in internal DB
  │     • Useful for infant mortality rate tracking internally
  │
  └─► If ADULT:
        • Enter citizen National ID → fetch from internal citizens table
        • Enter cause of death
        • Enter date of death
        • Link duty officer (doctor/nurse) from hospital_officers table
        • Citizen vital_status → DECEASED in citizens table
        • Certificate generated → stored in internal rita_certificates table
        • Individual / family receives printed or digital copy


────────────────────────────────────────────────────────────────────────────────
  2.9  MARRIAGE REGISTRATION FLOW  (by Village Officer)
────────────────────────────────────────────────────────────────────────────────

  STEP 1: Enter National IDs
          │  • Husband National ID → validate in internal citizens table
          │  • Wife National ID → validate in internal citizens table
          ▼
  STEP 2: Validate Marriage Rules (queried from internal marriages table)
          │  MALE:   Can have multiple wives simultaneously
          │  FEMALE: Can have only ONE husband at a time.
          │          Exception: allowed to register new marriage if
          │          last husband's vital_status = DECEASED
          │          or last marriage status = DISSOLVED / UNREGISTERED
          ▼
  STEP 3: Generate Marriage Certificate (Internal)
          │  • System assigns unique marriage_id
          │  • Marriage record inserted into internal marriages table
          │  • Certificate PDF generated, signed via govt PKI
          │  • Stored in rita_certificates table
          │  • Marriage status = ACTIVE
          │  • Individual receives printed or digital copy
          ▼
  [Marriage Record Created in Central Database — used for lineage tracking]


  UNREGISTER MARRIAGE FLOW:
  ──────────────────────────
  • Officer enters marriage_id (looked up from internal marriages table)
  • Couple provide reasons for marriage dissolution
  • Dissolution request submitted to authorized institution (RITA module)
  • RITA module reviewer checks reasons; approves or rejects internally
  • On approval:
    - Marriage status → DISSOLVED or UNREGISTERED
    - dissolution_date, dissolution_reason, dissolution_authority recorded
  • Wife's marital lock released → eligible to re-marry


────────────────────────────────────────────────────────────────────────────────
  2.10  MIGRATION REGISTRATION FLOW  (by Village Officer)
────────────────────────────────────────────────────────────────────────────────

  [Source Village Officer — citizen's current village]
          │
          ▼
  STEP 1: Officer enters citizen National ID
          │  • System queries internal citizens table
          │  • Returns current residence info (village, ward, district)
          ▼
  STEP 2: Enter Migration Details
          │  • New residence: Region → District → Ward → Village
          │    (all selected from internal geography tables)
          │  • Reason for migration:
          │    Economic activities / Job / Family / Education / Other
          ▼
  STEP 3: Migration Request Created in Internal DB
          │  • Record inserted into migrations table
          │  • Status: PENDING
          │  • Expiry: request_date + 30 days
          │  • Notification sent to TARGET village officer (via internal
          │    notification service + push notification)
          ▼
  [Citizen must physically visit target village within 30 days]
          │
          ▼
  STEP 4: Target Village Officer Receives Request
          │  • Sees pending migration request in app (from internal DB)
          │  • Verifies citizen identity in person
          ▼
  STEP 5: Confirm or Reject
          │
          ├─► CONFIRMED:
          │     • Citizens table: current_village_id updated to new village
          │     • Migration status → CONFIRMED
          │     • Population counts auto-updated in Redis counters
          │       and reflected in population_snapshots
          │
          ├─► REJECTED:
          │     • Migration status → CANCELLED
          │     • Reason recorded in migrations table
          │
          └─► NO ACTION within 30 days:
                • System auto-cancels (scheduled job)
                • Migration status → EXPIRED
                • Citizen remains at original village

  [Unchanged from v1.0 — all security controls remain the same.
   The only impact of the self-contained architecture on security:
   • No external API keys or mTLS certificates needed for NIDA/MoHCDGEC
   • Reduced attack surface (no outbound data flows for citizen PII)
   • All sensitive data stays within NBS network perimeter]


════════════════════════════════════════════════════════════════════════════════
  4.1  AUTHENTICATION SECURITY
════════════════════════════════════════════════════════════════════════════════

  PASSWORD HASHING
  • Algorithm: bcrypt with cost factor ≥ 12

  JSON WEB TOKENS (JWT)
  • Access token TTL:  15 minutes
  • Refresh token TTL: 7 days (rotated on each use)
  • Signed with RS256

  ONE-TIME LOGIN TOKENS
  • SHA-256 hash stored in DB (never raw token)
  • TTL: 24 hours, single-use

  MFA
  • TOTP (RFC 6238) or SMS OTP (6-digit, 5-min TTL)

  ACCOUNT PROTECTION
  • Lockout after 5 failed attempts → exponential backoff
  • Permanent lock requires admin reset


════════════════════════════════════════════════════════════════════════════════
  4.2  DEVICE BINDING & GEO-FENCE SECURITY
════════════════════════════════════════════════════════════════════════════════

  • HMAC-SHA256 device fingerprint on first activation
  • 1 device per officer — district admin reset required to change
  • ST_Within(officer_gps, geo_fence_polygon) on every session start
  • Geo-fence polygons sourced from internal DB (regions/villages tables)
  • Offline mode: polygon cached locally (encrypted)
  • Mock/spoofed GPS → BLOCKED via sensor fusion checks


════════════════════════════════════════════════════════════════════════════════
  4.3  DATA ENCRYPTION
════════════════════════════════════════════════════════════════════════════════

  IN TRANSIT:  TLS 1.3 + mTLS (service-to-service) + certificate pinning
  AT REST:     AES-256 tablespace + pgcrypto column-level for PII/biometrics
  KEY MGMT:    HSM / cloud KMS, automatic 90-day rotation


════════════════════════════════════════════════════════════════════════════════
  4.4  ROLE-BASED ACCESS CONTROL (RBAC)
════════════════════════════════════════════════════════════════════════════════

  ROLE               READ                          WRITE
  ─────────────────────────────────────────────────────────────────────────
  Super Admin        ALL tables, ALL regions        All (incl. user mgmt)
  District Admin     Own district only              Officer CRUD, facility reg
  Village Officer    Own village citizens only      Citizen, building, infra,
                                                    marriage, migration, death
  Hospital Officer   Own facility records only      Birth, death (facility)
  Public User        Aggregated/anonymized only     None (read-only)
  ─────────────────────────────────────────────────────────────────────────

  • Row-Level Security (RLS) at PostgreSQL level
  • Public app: ZERO PII exposed — aggregated counts only


════════════════════════════════════════════════════════════════════════════════
  4.5  AUDIT TRAIL
════════════════════════════════════════════════════════════════════════════════

  • Every INSERT, UPDATE, DELETE → audit_logs entry (append-only)
  • Captures: actor, action, target, old/new data, IP, GPS, timestamp


════════════════════════════════════════════════════════════════════════════════
  4.6  NETWORK SECURITY
════════════════════════════════════════════════════════════════════════════════

  • Admin dashboards: VPN or IP whitelist required
  • Rate limiting via Redis
  • DDoS: CDN/WAF layer
  • Outbound: only email/SMS gateway and govt PKI signing endpoint
  • IDS on all network segments
